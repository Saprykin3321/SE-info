Техническое задание на разработку системы управления заказами интернет-магазина

Версия 1.0 от 2026-03-08. Автор: Сапрыкин С.С. Описание изменений: Начальная версия.

1. Введение

1.1 Цель документа
Настоящий документ определяет функциональные и технические требования к разработке системы управления заказами (Order Management System, OMS) для интернет-магазина. Система предназначена для автоматизации процессов оформления, обработки и отслеживания заказов.

1.2 Область применения
Система используется клиентами интернет-магазина для оформления заказов, а также сотрудниками (менеджерами, операторами склада) для их обработки и выполнения.

1.3 Определения и сокращения
Термин: Заказ (Order). Определение: Сущность, содержащая информацию о покупке: список товаров, количество, сумма, статус, данные покупателя.
Термин: Покупатель (Customer). Определение: Зарегистрированный пользователь или гость, оформляющий заказ.
Термин: Статус заказа (Order Status). Определение: Текущее состояние заказа в процессе выполнения (новый, оплачен, передан в доставку, выполнен, отменён).
Термин: API. Определение: Интерфейс для взаимодействия между клиентской частью и бэкендом.

3. Требования к Backend

3.1 Описание базы данных

Таблица customers (покупатели)
Хранит информацию о зарегистрированных покупателях. Для гостевых заказов может создаваться запись с минимальными данными или использоваться отдельная логика (в текущей версии предполагается, что покупатель заполняет данные при каждом заказе, но для упрощения введена таблица customers).

Поле: id, Тип: SERIAL, Ограничения: PRIMARY KEY, Описание: Уникальный идентификатор покупателя, Пример: 1
Поле: full_name, Тип: VARCHAR(255), Ограничения: NOT NULL, Описание: Полное имя, Пример: "Иван Петров"
Поле: phone, Тип: VARCHAR(20), Ограничения: UNIQUE, NOT NULL, Описание: Номер телефона в международном формате, Пример: "+71234567890"
Поле: email, Тип: VARCHAR(255), Ограничения: UNIQUE, NOT NULL, Описание: Адрес электронной почты, Пример: "ivan@example.com"
Поле: address, Тип: TEXT, Ограничения: , Описание: Адрес доставки (может переопределяться в заказе), Пример: "г. Москва, ул. Ленина, д.1, кв.2"
Поле: created_at, Тип: TIMESTAMP, Ограничения: DEFAULT NOW(), Описание: Дата и время регистрации, Пример: "2025-03-08 12:00:00"

Индексы:
- phone (для быстрого поиска по телефону)
- email (для поиска по email)

Таблица orders (заказы)
Содержит основную информацию о заказе.

Поле: id, Тип: SERIAL, Ограничения: PRIMARY KEY, Описание: Уникальный номер заказа, Пример: 12345
Поле: customer_id, Тип: INTEGER, Ограничения: FOREIGN KEY (customers.id), Описание: Ссылка на покупателя, Пример: 1
Поле: status, Тип: VARCHAR(50), Ограничения: NOT NULL, DEFAULT 'new', Описание: Текущий статус заказа (см. перечень статусов), Пример: "new"
Поле: total_amount, Тип: DECIMAL(10,2), Ограничения: NOT NULL, Описание: Итоговая сумма заказа, Пример: 4500.50
Поле: payment_method, Тип: VARCHAR(50), Ограничения: NOT NULL, Описание: Способ оплаты ("card", "cash", "online"), Пример: "card"
Поле: created_at, Тип: TIMESTAMP, Ограничения: DEFAULT NOW(), Описание: Дата и время создания, Пример: "2025-03-08 12:00:00"
Поле: updated_at, Тип: TIMESTAMP, Ограничения: DEFAULT NOW() ON UPDATE NOW(), Описание: Дата и время последнего обновления, Пример: "2025-03-08 12:05:00"

Перечень статусов (статусная модель):
- new – новый заказ, ожидает оплаты/подтверждения
- paid – оплачен
- shipped – передан в доставку
- delivered – доставлен
- cancelled – отменён

Индексы:
- customer_id (для получения заказов конкретного покупателя)
- status (для фильтрации по статусу)
- created_at (для сортировки по дате)

Таблица order_items (позиции заказа)
Содержит товары, входящие в заказ, с ценой на момент заказа.

Поле: id, Тип: SERIAL, Ограничения: PRIMARY KEY, Описание: Уникальный идентификатор позиции, Пример: 1
Поле: order_id, Тип: INTEGER, Ограничения: FOREIGN KEY (orders.id) ON DELETE CASCADE, Описание: Ссылка на заказ, Пример: 12345
Поле: product_id, Тип: INTEGER, Ограничения: FOREIGN KEY (products.id), Описание: Идентификатор товара, Пример: 101
Поле: quantity, Тип: INTEGER, Ограничения: NOT NULL, CHECK (quantity > 0), Описание: Количество единиц, Пример: 2
Поле: price, Тип: DECIMAL(10,2), Ограничения: NOT NULL, Описание: Цена одной единицы на момент заказа, Пример: 1500.50

Уникальное ограничение: уникальность пары (order_id, product_id) гарантирует, что один товар не будет добавлен в заказ дважды.

Индексы:
- order_id (для быстрого получения всех позиций заказа)

Таблица products (товары)
Справочник товаров с актуальной информацией.

Поле: id, Тип: SERIAL, Ограничения: PRIMARY KEY, Описание: Уникальный идентификатор товара, Пример: 101
Поле: name, Тип: VARCHAR(255), Ограничения: NOT NULL, Описание: Название товара, Пример: "Смартфон Galaxy S21"
Поле: description, Тип: TEXT, Ограничения: , Описание: Описание товара, Пример: "128 ГБ, черный"
Поле: price, Тип: DECIMAL(10,2), Ограничения: NOT NULL, Описание: Текущая цена, Пример: 50000.00
Поле: stock, Тип: INTEGER, Ограничения: NOT NULL, CHECK (stock >= 0), Описание: Остаток на складе, Пример: 10
Поле: created_at, Тип: TIMESTAMP, Ограничения: DEFAULT NOW(), Описание: Дата добавления, Пример: "2025-03-01 10:00:00"

Индексы:
- name (для поиска по названию)

3.2 Описание API эндпоинтов

Метод: GET, Эндпоинт: /api/orders/, Описание: Получить список заказов (с фильтрацией), Доступ: Менеджер, администратор
Метод: GET, Эндпоинт: /api/orders/{id}/, Описание: Получить детальную информацию о заказе, Доступ: Покупатель (свой), менеджер
Метод: POST, Эндпоинт: /api/orders/, Описание: Создать новый заказ, Доступ: Покупатель
Метод: PATCH, Эндпоинт: /api/orders/{id}/status, Описание: Изменить статус заказа, Доступ: Менеджер, оператор

3.2.1 GET /api/orders
Описание: Возвращает список заказов с возможностью фильтрации, пагинации и сортировки. Используется менеджерами и администраторами для просмотра всех заказов, а также покупателями (только свои заказы) – в зависимости от роли.
URL: /api/orders
Метод: GET
Аутентификация: Требуется JWT-токен в заголовке Authorization: Bearer <token>

Параметры запроса (указываются в query):
Параметр: status, Тип: string, Обязательный: нет, По умолчанию: –, Описание: Фильтр по статусу заказа (например, new, paid)
Параметр: customer_id, Тип: integer, Обязательный: нет, По умолчанию: –, Описание: Фильтр по идентификатору покупателя (для админов)
Параметр: date_from, Тип: string, Обязательный: нет, По умолчанию: –, Описание: Начальная дата создания (в формате YYYY-MM-DD)
Параметр: date_to, Тип: string, Обязательный: нет, По умолчанию: –, Описание: Конечная дата создания (в формате YYYY-MM-DD)
Параметр: page, Тип: integer, Обязательный: нет, По умолчанию: 1, Описание: Номер страницы для пагинации
Параметр: limit, Тип: integer, Обязательный: нет, По умолчанию: 20, Описание: Количество записей на странице (макс. 100)
Параметр: sort_by, Тип: string, Обязательный: нет, По умолчанию: "created_at", Описание: Поле для сортировки (id, created_at, total_amount)
Параметр: order, Тип: string, Обязательный: нет, По умолчанию: "desc", Описание: Направление сортировки (asc или desc)

Пример запроса:
GET /api/orders?status=paid&date_from=2025-03-01&page=2&limit=10&sort_by=total_amount&order=desc
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

Логика работы:
1. Проверка прав доступа
   - Если роль пользователя – admin или manager, возвращаются заказы всех покупателей (с учётом фильтров).
   - Если роль – customer, параметр customer_id игнорируется или принудительно устанавливается равным ID текущего покупателя, чтобы он видел только свои заказы. При попытке передать чужой customer_id возвращается ошибка 403.
2. Применение фильтров
   - status – точное совпадение со значением в таблице orders.
   - customer_id – точное совпадение (доступно только для менеджеров/админов).
   - date_from и date_to – фильтрация по полю created_at (включая обе границы). Даты преобразуются в диапазон с 00:00:00 до 23:59:59.
3. Пагинация
   - Вычисляется OFFSET = (page - 1) * limit.
   - В ответе возвращается мета-информация с общим количеством записей и количеством страниц.
4. Сортировка
   - Поле сортировки (sort_by) должно быть одним из допустимых: id, created_at, total_amount. При недопустимом значении используется created_at по умолчанию.
   - Направление order может быть asc или desc.
5. Формирование ответа
   - Для каждого заказа выбираются поля из таблицы orders.
   - Дополнительно подгружаются данные покупателя (full_name, phone, email) из таблицы customers.
   - Также для каждого заказа можно включить массив позиций (items), но по умолчанию он не включается для сокращения объёма данных. При необходимости можно добавить параметр include_items=true (не входит в базовую спецификацию).
6. Оптимизация
   - Используются индексы по полям status, customer_id, created_at для ускорения фильтрации и сортировки.
   - Запрос выполняется с JOIN к таблице customers для получения имени покупателя.
